# Phase 3.2 – Speech Pipeline: STT + TTS Loop

## 1. Objective

Phase 3.2 adds the speech processing layer on top of the LiveKit infrastructure established in Phase 3.1. The goal is to prove that the Node.js voice agent can speak a greeting into the LiveKit room via Sarvam TTS, capture the user's spoken response via `AudioStream` frame accumulation, send the captured audio to Sarvam STT for transcription, and echo the transcript back — completing a full STT → TTS round-trip within a single LiveKit room.

No MongoDB interaction, job creation, or worker matching occurs. This phase is strictly speech: audio capture, format conversion, API integration, and audio playback.

## 2. Architecture Implemented

```
Browser Simulator                         LiveKit Cloud
(livekit-client CDN)  ── WebRTC ──────>  (Free Tier)
       ^                                      |
       |                                WebRTC (agent)
       |                                      |
       |                                      v
       |                              Express.js Voice Agent
       |                              (@livekit/rtc-node)
       |                                      |
       |                         +------------+------------+
       |                         |                         |
       |                    Sarvam STT               Sarvam TTS
       |                   (saaras:v3)              (bulbul:v3)
       |                    Ears                      Mouth
       |                         |                         |
       |                         +------------+------------+
       |                                      |
       +──────────── TTS audio published ─────+
```

**Speech pipeline flow:**

1. Agent connects to LiveKit room (reuses Phase 3.1 infrastructure).
2. Agent calls `sarvamService.textToSpeech()` with the greeting text.
3. The returned WAV buffer is parsed, upsampled from 16kHz to 48kHz, and published to the room as a `LocalAudioTrack` via `AudioSource.captureFrame()`.
4. Agent waits for the remote participant's audio track via `_waitForAudioTrack()`.
5. Once a track is subscribed, `_listen()` creates an `AudioStream` and accumulates `Int16Array` frames for 8 seconds.
6. Captured frames are downmixed to mono (if multi-channel), resampled from 48kHz to 16kHz, and wrapped in a 44-byte WAV header.
7. The WAV buffer is sent to Sarvam STT (`POST /speech-to-text`), which returns a transcript.
8. The transcript is prefixed with "Aapne kaha: " and spoken back via TTS — completing the echo loop.
9. Agent disconnects cleanly after the echo.

**Audio format pipeline:**

```
LiveKit Room Audio (48kHz, potentially stereo)
    → downmixToMono() → Int16Array (mono)
    → resample(48000, 16000) → Int16Array (16kHz mono)
    → createWavBuffer() → Buffer (WAV, 44-byte header + PCM)
    → Sarvam STT API
    → transcript string

Sarvam TTS API → base64 WAV (16kHz)
    → parseWav() → PCM + metadata
    → resample(16000, 48000) → Int16Array (48kHz mono)
    → AudioFrame chunks (480 samples = 10ms each)
    → AudioSource.captureFrame() → published to LiveKit room
```

## 3. Key Design Decisions

**WAV encoding/decoding handled in-process.**
Rather than depending on an external library (e.g., `wav` npm package), `sarvam.service.js` implements `createWavBuffer()` and `parseWav()` directly. The 44-byte RIFF/WAVE header is constructed manually. This eliminates a dependency and keeps the audio path fully transparent for debugging.

**Integer-ratio resampling for 48kHz ↔ 16kHz.**
LiveKit rooms default to 48kHz audio. Sarvam STT requires 16kHz WAV input. The `resample()` method uses simple integer decimation (every 3rd sample for 48→16) and sample repetition (3× for 16→48). This avoids the complexity of a polyphase resampler while remaining adequate for speech. A linear interpolation fallback is included for non-integer ratios.

**Stereo-to-mono downmix by channel averaging.**
`downmixToMono()` averages interleaved channel samples. This handles stereo and arbitrary channel counts (tested with 4-channel input). Mono input passes through without allocation.

**Real-time pacing during TTS playback.**
Each 480-sample frame (10ms at 48kHz) is fed to `AudioSource.captureFrame()` with a matching `_delay(10)` between frames. Without pacing, all frames would be queued instantly, causing audio distortion or dropped frames on the receiving end.

**Graceful failure at every API boundary.**
Both `speechToText()` and `textToSpeech()` wrap their `axios.post()` calls in try/catch with explicit handling for `ECONNABORTED` (timeout) and network errors. STT failures return `{ transcript: '', languageCode: null }`. TTS failures return `null`. The pipeline checks each return value and falls back to a spoken error message rather than crashing.

**10-second API timeout with AbortController.**
Both STT and TTS calls use `AbortController` with a 10-second timeout. The controller aborts the request if the API doesn't respond in time. This prevents the agent from hanging indefinitely on a slow or unresponsive Sarvam endpoint.

**Track publish/unpublish lifecycle.**
Each `_speak()` call creates a fresh `LocalAudioTrack`, publishes it, feeds all frames, waits 200ms for the final frame to flush, then unpublishes. This prevents track stacking (multiple simultaneous audio tracks) and ensures clean audio in the room.

## 5. Service APIs Added

### SarvamService

#### `speechToText(wavBuffer, options?)` → `{ transcript: string, languageCode: string|null }`

Sends a WAV buffer to Sarvam STT (`POST https://api.sarvam.ai/speech-to-text`) as multipart form data. Returns the transcript and detected language. Returns `{ transcript: '', languageCode: null }` on failure.

**Options:**
- `languageCode` (default: `'hi-IN'`) — hint for the STT model.
- `model` (default: `'saaras:v3'`) — Sarvam STT model.

#### `textToSpeech(text, options?)` → `Buffer | null`

Sends text to Sarvam TTS (`POST https://api.sarvam.ai/text-to-speech`). Returns a WAV `Buffer` decoded from the base64 response. Returns `null` on failure.

**Options:**
- `languageCode` (default: `'hi-IN'`)
- `speaker` (default: `'shubh'`)
- `sampleRate` (default: `16000`)

#### `createWavBuffer(pcmBuffer, sampleRate, numChannels)` → `Buffer`

Constructs a complete WAV file from raw PCM Int16 data with a 44-byte RIFF header.

#### `parseWav(wavBuffer)` → `{ pcm, sampleRate, numChannels, bitsPerSample }`

Extracts PCM data and metadata from a WAV buffer. Searches for the `data` chunk marker for safety.

#### `resample(samples, fromRate, toRate)` → `Int16Array`

Integer-ratio resampling: decimation for downsampling, repetition for upsampling, linear interpolation for non-integer ratios. Returns the same reference if rates are equal.

#### `downmixToMono(samples, numChannels)` → `Int16Array`

Averages interleaved multi-channel samples into mono. Returns the same reference for mono input.

### VoiceAgentService (New Methods)

#### `_runPipeline(roomName, room)`

Orchestrates the full speech loop: greet → wait for audio track → listen → STT → echo → disconnect. Called in the background by `startVoiceSession()`. Handles all errors internally without propagating to the caller.

#### `_speak(room, text)`

Converts text to audio via Sarvam TTS, parses the WAV, upsamples to 48kHz, creates an `AudioSource` and `LocalAudioTrack`, publishes to the room, feeds frames with real-time pacing (10ms per 480-sample frame), then unpublishes. Skips silently if TTS returns null.

#### `_listen(remoteTrack, durationMs)` → `Buffer | null`

Creates an `AudioStream` on the remote track, accumulates frames for `durationMs` milliseconds, downmixes to mono, resamples to 16kHz, and returns a complete WAV buffer. Returns `null` if no frames are captured.

#### `_waitForAudioTrack(room, timeoutMs)` → `Track | null`

Checks existing remote participants for an audio track. If none found, listens for `TrackSubscribed` events with a timeout. Returns `null` if no audio track arrives within the timeout.

**Mocking:** All external dependencies are mocked (axios, form-data, @livekit/rtc-node, livekit-server-sdk, mongoose, bcryptjs, jsonwebtoken, uuid). Zero network calls or LiveKit connections during testing.

## 7. Known Limitations

- **No MongoDB interaction.** No jobs are created, no workers are matched, no user context is loaded. Deferred to Phase 3.3.
- **No LLM integration.** Responses are simple echoes, not conversational. Deferred to Phase 3.3/3.4.
- **No intent extraction wired.** `extractIntent()` is planned in the design docs but not yet implemented in this phase. The echo loop validates the STT → TTS round-trip without requiring intent parsing.
- **Integer-ratio resampling only.** The `resample()` method works correctly for 48kHz ↔ 16kHz (ratio 3) but uses a basic linear interpolation fallback for non-integer ratios. Not suitable for arbitrary sample rate conversion.
- **No Voice Activity Detection (VAD).** The agent listens for a fixed 8-second window regardless of whether the user is speaking. Silence is captured and sent to STT, which may return an empty transcript.
- **Single-turn conversation.** The agent speaks a greeting, listens once, echoes, and disconnects. Multi-turn dialogue is not implemented.
- **No authentication on voice endpoints.** The `/api/voice/sessions/initiate` endpoint does not require auth tokens. Acceptable for hackathon; must be addressed for production.
- **TTS playback pacing uses `setTimeout`.** The 10ms delay between frames relies on JavaScript timer precision, which can drift under load. Acceptable for demo use.
- **Agent identity includes timestamp.** Each session uses `node-agent-{Date.now()}` as the identity, which avoids collisions but makes identity tracking harder across logs.

## 9. Readiness for Phase 3.3

Phase 3.2 provides the complete speech pipeline that Phase 3.3 depends on:

- **STT is proven.** `sarvamService.speechToText()` reliably transcribes Hindi/English audio. Phase 3.3 will pipe transcripts into intent parsing and MongoDB lookups.
- **TTS is proven.** `sarvamService.textToSpeech()` reliably converts text to room-playable audio. Phase 3.3 will use it to speak job briefs with dynamic worker/job context injected from MongoDB.
- **Audio format pipeline is complete.** Downmixing, resampling, WAV encoding/decoding all work end-to-end. No audio format changes are needed for Phase 3.3.
- **`_speak()` and `_listen()` are reusable.** Phase 3.3 will call these same methods inside `runWorkerFlow()` and `runClientFlow()` without modification.
- **Error handling is production-grade.** All failure paths return safe defaults. Phase 3.3 can add MongoDB context loading without worrying about speech layer crashes.
- **`_waitForAudioTrack()` with timeout is ready.** Phase 3.3's worker flow will reuse this to detect when the worker's browser simulator connects.

Phase 3.3 scope: wire `voiceAgent.service.js` to MongoDB (`User` and `Job` models), implement `runWorkerFlow()` to brief a worker and collect an Accept/Decline decision, and update `job.status` and `user.status` based on the spoken response.
