# Phase 3.1 – LiveKit Infrastructure Layer

## 1. Objective

Phase 3.1 establishes the real-time audio transport layer for the Voice Dispatch System. The goal is to prove that a browser-based simulator and a server-side Node.js agent can both connect to a shared LiveKit Cloud room, confirm mutual presence, and exchange audio tracks over WebRTC.

No speech processing, language model interaction, or database-driven job logic is included. This phase is strictly infrastructure: room creation, token generation, participant connection, and lifecycle management.

## 2. Architecture Implemented

```
Browser Simulator                         LiveKit Cloud
(livekit-client CDN)  ── WebRTC ──────>  (Free Tier)
                                              |
                                         WebRTC (token B)
                                              |
                                              v
                                        Express.js Server
                                        (@livekit/rtc-node)
```

**Request flow:**

1. The browser calls `POST /api/voice/sessions/initiate` with an optional `roomName`.
2. Express generates a JWT access token for the simulator via `livekit-server-sdk`.
3. Express fires `voiceAgentService.startVoiceSession(roomName)` as a non-blocking background task.
4. The response returns immediately with the simulator token, room name, and LiveKit WebSocket URL.
5. The browser connects to LiveKit Cloud using the token.
6. The Node agent connects to the same room using a separately generated token via `@livekit/rtc-node`.
7. Both participants appear in the LiveKit room. Audio tracks can be published and subscribed.

**Token generation** uses `livekit-server-sdk`'s `AccessToken` class with `roomJoin`, `canPublish`, and `canSubscribe` grants. Tokens have a 1-hour TTL.

**Agent startup** is fire-and-forget. The HTTP response is not blocked on the agent's connection result. If the agent fails to connect, the error is logged server-side but the client still receives a valid token.

## 3. Key Design Decisions

**Non-blocking agent startup.**
The agent connection is launched via a `.catch()`-guarded promise. The HTTP response returns immediately. This prevents slow or failed agent connections from blocking the client.

**Graceful degradation when `@livekit/rtc-node` is unavailable.**
The native RTC module is loaded via dynamic `await import()` inside a try/catch. If the native binary is missing (common on Windows without build tools), the service falls back to "token-only" mode: tokens are generated and returned, but the agent does not join the room. The server continues to run.

**Early environment variable loading.**
`server.js` uses `import 'dotenv/config'` as its first import statement rather than calling `dotenv.config()` after other imports. This is required because ES module imports are hoisted and executed before any runtime code. Without this fix, `livekit.service.js` would attempt to read environment variables before they are loaded, causing a startup crash.

**No MongoDB interaction.**
Phase 3.1 is deliberately isolated from the job and user data layer. This allows the infrastructure to be tested independently and reduces the surface area for debugging connection issues.

## 5. API Surface Added

### POST /api/voice/sessions/initiate

Creates a LiveKit room, generates a browser token, and starts the Node agent.

**Request:**
```json
{
  "roomName": "dispatch-room-42",
  "simulatorIdentity": "client-phone"
}
```

Both fields are optional. If omitted, `roomName` defaults to `room-{timestamp}` and `simulatorIdentity` defaults to `simulator-{timestamp}`.

**Response (200):**
```json
{
  "roomName": "dispatch-room-42",
  "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6...",
  "livekitUrl": "wss://parvanamai-x8qo8jod.livekit.cloud",
  "agentIdentity": "node-agent",
  "simulatorIdentity": "client-phone"
}
```

### GET /api/voice/status

Returns LiveKit credential health and active session count.

**Response (200):**
```json
{
  "phase": "3.1",
  "livekit": { "healthy": true },
  "activeSessions": 0,
  "env": {
    "LIVEKIT_URL": "set",
    "LIVEKIT_API_KEY": "set",
    "LIVEKIT_API_SECRET": "set"
  }
}
```

### DELETE /api/voice/sessions/:roomName

Manually disconnects the agent from the specified room and cleans up the session timer.

**Response (200):**
```json
{
  "message": "Session dispatch-room-42 ended"
}
```

## 7. Known Limitations

- **No speech pipeline.** STT and TTS integration is deferred to Phase 3.2.
- **No MongoDB interaction.** No jobs are created, no workers are matched.
- **Agent identity is hardcoded.** All sessions use `node-agent` as the agent identity. Concurrent sessions in the same room would cause identity collisions.
- **Room name collision at millisecond resolution.** Auto-generated room names use `Date.now()`, which can collide under rapid concurrent requests.
- **No authentication middleware on voice endpoints.** Any caller can create sessions. Acceptable for hackathon testing; must be addressed before production.
- **No rate limiting.** Endpoints are unprotected against request flooding.
- **No input sanitization on `roomName`.** Arbitrary strings are passed directly to LiveKit.
- **Session timeout uses a fixed 2-minute timer.** No configurable or adaptive timeout.

## 8. Completion Criteria (Met)

- LiveKit Cloud room tokens are generated with correct `roomJoin`, `canPublish`, and `canSubscribe` grants.
- Browser simulator connects to LiveKit and publishes microphone audio.
- Node agent connects to the same room as a server-side participant.
- Both participants appear in the LiveKit Cloud dashboard.
- Agent startup does not block the HTTP response.
- Server starts without crashing when `@livekit/rtc-node` native module is unavailable.
- Environment variables are loaded before any service constructor executes.
- Health check endpoint validates LiveKit credentials at runtime.
- All 13 integration tests pass with zero external dependencies.

## 9. Readiness for Phase 3.2

Phase 3.1 provides the transport layer that Phase 3.2 depends on:

- **Room connectivity is proven.** The agent reliably joins rooms and subscribes to remote audio tracks.
- **Token generation is tested.** The same `createRoomToken()` function will be used to generate tokens for STT/TTS participants.
- **Event hooks are in place.** `RoomEvent.TrackSubscribed` is already wired in `voiceAgent.service.js`. Phase 3.2 will pipe subscribed audio buffers into Sarvam STT.
- **Session lifecycle is managed.** Rooms are created, tracked in a Map, and cleaned up on disconnect or timeout. Phase 3.2 will extend this with speech state tracking.
- **The service structure is established.** `voiceAgent.service.js` will grow to include STT/TTS orchestration without requiring architectural changes.

Phase 3.2 scope: integrate Sarvam AI STT on incoming audio, generate TTS responses, and close the speech loop within the LiveKit room.
